# 项目整体策划与想法

> 整理时间：2026-05-05

> 用途：记录项目核心想法、产品逻辑、技术方案，作为后续开发的依据

## 一、产品定位

### 一句话描述

**面向大学生的竞赛全流程 AI 工作台**：从选赛出发，根据比赛类型自动分流至「方案赛道」或「竞技赛道」，全流程 AI 辅助。

### 最终产品形态

网页 / 微信小程序 / APP（三端），用户打开即用，不需要安装任何工具。

### 当前阶段

比赛展示阶段：验证核心流程能跑通 + 搭建网页框架（架子），商业化部分写在策划书里即可。

## 二、核心流程：双赛道架构

### 为什么需要双赛道？

不是所有比赛都需要写计划书和 PPT。像蓝桥杯、大英赛这类比赛，核心是上机写代码或笔试做题，评判标准是解题正确率而非方案质量。因此选赛后必须按比赛类型分流：

```
                    选赛（共用入口）
                          │
                          ▼
              比赛类型判断（系统自动）
                          │
           ┌──────────────┴──────────────┐
           ▼                              ▼
      📄 方案赛道                      ⚡ 竞技赛道
     (文档产出类)                     (技能考核类)
           │                              │
    调研 → 方案 → PPT                  备赛规划 → 学习 → 训练
           │                              │
           └──────────┬───────────────────┘
                      ▼
                 赛后沉淀（共用）
```

### 两赛道核心差异

| 维度 | 方案赛道 | 竞技赛道 |
|------|---------|---------|
| 典型比赛 | 互联网+、大创、商赛、挑战杯 | 蓝桥杯、ACM、大英赛、数模 |
| 核心产出 | 商业计划书、申报书、PPT | 解题能力、代码、知识点掌握 |
| AI辅助方式 | 写作引导、框架搭建、审改迭代 | 知识点梳理、刷题路径规划、错题分析 |
| 用户需求 | 「帮我写出一份好方案」 | 「帮我系统地学会这个技能」 |
| 工作模式 | 调研→写→审→改 | 学→练→测→复 |

### 灰色地带比赛的处理

部分比赛处于两赛道之间的灰色地带：

| 比赛 | 归属赛道 | 处理方式 |
|------|:---:|------|
| 数学建模 | 竞技赛道 | 核心是建模解题能力，论文只是解题结果的呈现形式。AI 辅助重点是建模思路、算法选择、代码调试 |
| 电子设计/机械创新 | 竞技赛道 | 核心是技术能力+实物制作，但有设计报告需求。归竞技赛道，额外标记「需写报告」 |
| 外研社演讲/辩论/模联 | 方案赛道 | 核心是文稿准备（AI 能帮），现场表现部分标注为「辅助训练」 |

### 方案赛道模块详情

> 以下为方案赛道（文档产出类竞赛）的完整流程模块。

### 模块1：选赛（个性化竞赛推荐）—— 两赛道共用

**功能描述：**
大学生输入自己的个人信息（学校、专业、年级、兴趣、目标等），系统自动生成个性化的竞赛推荐，帮助他选择适合的竞赛。

**推荐维度：**

- 保研加分类竞赛
- 考研相关竞赛
- 就业导向竞赛
- 能力提升类竞赛

**输出内容：**
竞赛名称、匹配理由、参赛建议、同类竞赛横向对比

**当前进度：** ✅ V1.7 已跑通核心逻辑（match_competitions.py）

**后续优化方向：**

- 画像维度扩展（GPA区间、已有获奖经历、技术栈）
- 增加「同类竞赛横向对比表」
- 订阅功能：新竞赛出现时主动推送

---

### 模块2：调研（方案赛道）

**功能描述：**
确定目标竞赛后，进行信息收集和分析。

**子步骤：**

#### 2.1 往届获奖作品收集

- **网络收集**：AI 自动搜索网上公开的往届获奖信息
- **用户补充**：引导用户自己收集线下资料（找学长、问老师），因为公开完整项目书极少
- **AI 分析**：结合网络信息和用户提供的资料，给出分析建议
- **兜底机制**：用户无法收集时，AI 基于网络可获取信息给出最优分析（标注 ⚠️ 待用户确认）

#### 2.2 市场/行业调研

- 分析目标领域是否已有人做过类似项目
- 竞品分析、同赛道竞争格局
- 寻找差异化切入点

**关键风险：**
往届获奖作品公开信息极少，产品需管理用户预期——承诺「基于可获取信息给出最优分析」，而非「找到完整往届作品」。

---

### 模块3：方案设计（方案赛道）⭐ 核心差异化模块

**功能描述：**
这是整个产品最核心、最能体现差异化价值的模块。

**子步骤：**

#### 3.1 搭框架

- 结合用户想法，通过多轮沟通了解用户意图
- 基于往届获奖作品的结构搭建框架（不是凭空捏造，有据可依）
- 产出：结构化的项目方案大纲

#### 3.2 写初稿

- AI 给出框架后，用户与 AI 协作完成初稿
- 用户需要花心思参与，AI 辅助填充内容
- 分章节逐步完成，不一次性生成全文

#### 3.3 AI 扮演评审，反向审问

- 同一个 AI 先帮你写，再扮演评委挑毛病
- 比单纯让 AI「夸你」有用得多
- 模拟真实答辩前的准备过程

#### 3.4 审改迭代循环

- 「审 → 改 → 再审 → 再改」循环打磨
- 直至方案质量达标

**退出条件（待设定）：**

- 质量阈值：AI 评审打分达到某个分数
- 次数上限：最多迭代 N 轮后提示人工确认
- 用户主动退出：随时说「可以了」

**核心壁垒：**

- 精心设计的提示词（每个子步骤有对应提示词）
- 独特的流程方法论（什么时候调用哪个提示词、提示词之间如何衔接）
- 基于往届获奖作品结构的框架模板库

---

### 模块4：PPT 路演（方案赛道）

**功能描述：**
辅助用户准备 PPT 和路演，但不做完整 PPT 生成。

**子步骤：**

- PPT 大纲建议
- 答辩 Q&A 预判（评委可能会问什么、怎么回答）
- 路演话术建议

**定位：**
轻量级辅助模块。产品核心是「帮大学生做好竞赛项目」，不是「帮大学生做 PPT」。PPT 和路演更依赖设计感和个人风格，AI 辅助而非替代。

---

### 模块5：赛后沉淀（两赛道共用）⭐ 增长飞轮

**功能描述：**
项目结束后，AI 读取项目文件夹所有文件，自动总结和提取经验。

**子步骤：**

- 总结做了什么、遇到什么困难、怎么解决的
- 提炼可迁移的方法论
- 标记哪些环节做得好、哪些可以改进
- 整理经验库存入个人信息库

**长期价值：**
下一次开新项目时，这些经验自动被引用。越用越好用，形成增长飞轮——用户切换成本越来越高，形成壁垒。

**技术可行性：** 高。本质是「读文件 + LLM 总结 + 写 memory」。

### 竞技赛道模块详情

> 以下为竞技赛道（技能考核类竞赛）的完整流程模块。与方案赛道不同，竞技赛道的核心是「帮助用户系统地学会技能」而非「写文档」。

### 竞技模块1：备赛规划

**功能描述：**
确定目标竞赛后，AI 帮助用户制定系统的备赛计划。

**子步骤：**

- **学习路线图**：基于竞赛大纲和往年考点，规划从零到参赛的完整学习路径，按知识模块排列，标注优先级和预估耗时
- **资源清单**：推荐教材、视频教程、在线题库、刷题网站的链接和特点对比（哪些适合入门、哪些适合冲刺）
- **时间规划表**：根据距离考试的剩余时间，倒排每日/每周学习任务。支持「时间紧迫模式」（如只剩2周）和「充裕模式」（如还有3个月）

**输出示例：**

```
蓝桥杯备赛规划（剩余 45 天）
├── 第1-7天：基础算法回顾（排序、搜索、贪心）
├── 第8-14天：数据结构强化（栈、队列、树、图）
├── 第15-21天：常考题型专项（DP、回溯、二分）
├── 第22-28天：历年真题刷题（分类刷）
├── 第29-35天：模拟考试（计时、查漏补缺）
├── 第36-42天：错题重刷 + 薄弱点攻坚
└── 第43-45天：全真模拟 + 心态调整
```

---

### 竞技模块2：知识学习

**功能描述：**
AI 帮用户系统整理知识点，按模块拆分学习内容。

**子步骤：**

- **知识点梳理**：根据竞赛考纲，生成结构化的知识点清单，标注重要程度（必考/高频/低频）
- **AI 讲解**：用户对某个知识点不理解时，AI 用通俗语言讲解，配合例题帮助理解。可以追问、追问、再追问，直到懂为止
- **思维导图**：自动生成知识体系思维导图，帮助用户建立整体框架（非图片，用 Markdown 列表或 Mermaid 图）

**定位：**
AI 替代的不是「教材和老师」，而是「身边随时能问的那个人」——不懂就追问，AI 不会不耐烦。

---

### 竞技模块3：刷题训练

**功能描述：**
系统化的刷题辅助，不只做题还要分析。

**子步骤：**

- **题目分类汇总**：按知识模块和难度分级的题目清单，标注题目来源（LeetCode/洛谷/PTA 等）、题号、推荐刷题顺序
- **AI 讲解思路**：用户拿到一道题，AI 不直接给答案，先引导用户说思路 → AI 指出问题 → 用户修正 → AI 确认 → 再给最优解作为参考答案（启发式辅导，不是直接抄答案）
- **错题本**：自动记录做错的题、错因分类（思路错误/细节错误/知识点盲区）、同类题推荐
- **代码解答归档**：用户写的正确代码自动归档到 `06-代码/` 文件夹，按题型分类

**核心原则：**
AI 是陪练不是代练。不直接给答案，引导用户自己思考。

---

### 竞技模块4：模拟实战

**功能描述：**
考前模拟真实考试环境，检验准备效果。

**子步骤：**

- **模考记录**：AI 推荐一套模拟题（或用户自选），计时完成，记录每道题用时和得分。多次模考形成趋势曲线
- **正确率统计**：按知识模块统计正确率，自动识别薄弱点（如「动态规划正确率仅 40%」），生成薄弱点诊断报告
- **时间管理分析**：分析用户在各题型上的时间分配是否合理，给出考场时间策略建议

**输出示例：**

```
第三次模考总结（蓝桥杯模拟）
├── 总分：85/150（比上次 +12 分 ↑）
├── 模块正确率：
│   ├── 基础算法 92% ✅
│   ├── 数据结构 78% ⚠️
│   └── 动态规划 40% ❌ ← 薄弱点，建议回炉
├── 时间分析：填空题耗时过长（平均 12min/题），建议控制在 8min 内
└── 下次模考重点：DP 专项训练
```

---

### 竞技模块5：赛后复盘（与方案赛道共用模块5）

与方案赛道共用同一套赛后沉淀机制。竞技类比赛额外复盘内容：
- 哪些知识点考试遇到了但没准备到
- 刷题策略是否有效（刷题量 vs 实际考试覆盖度）
- 时间管理是否到位
- 下一次同类比赛的建议调整

## 二-补充：需求标签系统

### 为什么需要需求标签

两赛道能覆盖约 90% 的比赛，但有些比赛是混合型的（既有文档需求又有技能需求）。与其把赛道越拆越细，不如保持两赛道 + 给每个比赛打上**需求标签**，让系统精确知道该帮用户做什么。

### 标签体系

| 需求标签 | 含义 | 触发动作 |
|---------|------|---------|
| `商业计划书` | 需要写商业计划书 | 启动方案赛道模块3 |
| `申报书` | 需要填写申报书 | 启动方案赛道模板辅助 |
| `PPT路演` | 需要准备路演 PPT | 启动方案赛道模块4 |
| `答辩Q&A` | 需要准备答辩 | 方案赛道模块4 子步骤 |
| `算法学习` | 需要学编程算法 | 启动竞技赛道模块2+3 |
| `刷题训练` | 需要大量刷题 | 启动竞技赛道模块3 |
| `模拟考试` | 需要模拟考试 | 启动竞技赛道模块4 |
| `编程求解` | 需要写代码解题 | 创建 `06-代码/` |
| `数学建模` | 需要建模能力 | 竞技赛道模块2 数学方向 |
| `论文写作` | 需要写论文（如数模论文） | 额外触发方案赛道写作辅助 |
| `实物制作` | 需要制作实物 | 标注提醒（AI无法替代） |
| `设计报告` | 需要写设计报告 | 轻量方案辅助 |
| `演讲/辩论` | 需要现场表现 | 方案赛道辅助 + 标注「AI能力有限」 |

### 标签→赛道→文件夹映射示例

```
互联网+ (大创)
├── 赛道: 方案赛道
└── 标签: [商业计划书, 申报书, PPT路演, 答辩Q&A]
    → 全套方案赛道流程

蓝桥杯
├── 赛道: 竞技赛道
└── 标签: [算法学习, 刷题训练, 模拟考试]
    → 竞技赛道完整流程，创建 06-代码/

大英赛
├── 赛道: 竞技赛道
└── 标签: [模拟考试]
    → 竞技赛道流程，不创建 06-代码/

数学建模
├── 赛道: 竞技赛道
└── 标签: [数学建模, 编程求解, 论文写作]
    → 竞技流程 + 方案赛道写作辅助 + 06-代码/ + 07-数据/

电子设计竞赛
├── 赛道: 竞技赛道
└── 标签: [实物制作, 设计报告]
    → 竞技流程 + 轻量方案辅助 + 标注实物部分 AI 无法替代
```

### 设计原则

- **赛道决定大方向，标签决定具体做什么**
- 标签是系统预设的（非用户自定义），每个比赛初始化时自动匹配
- 灰色地带比赛不用纠结分类——标签直接写明需要什么，系统按标签触发对应模块

## 二-补充：双赛道文件夹模板方案

### 设计思路

系统初始化项目时，根据赛道类型 + 需求标签，自动创建对应文件夹。不做 AI 动态预测生成（容易出错），而是**提前写好模板，AI 只负责按规则匹配**。

### 方案赛道文件夹模板（当前工作区已实现）

```
项目根目录/
├── 项目信息.md          # 竞赛基本信息（系统维护）
├── 进度追踪.md          # 阶段 checklist（系统维护）
├── 项目日志.md          # 关键事件记录（系统自动追加）
├── 用户画像.md          # 用户基础信息 + 项目层信息
├── CLAUDE.md           # AI 工作规则
├── 01-调研/            # 往届作品分析、市场调研、竞品分析
├── 02-方案/            # 商业计划书、申报书、AI评审反馈、各版本迭代
├── 03-素材/            # 数据图表、参考文献、截图资料
├── 04-PPT/             # PPT大纲、答辩Q&A、路演话术
└── 05-复盘/            # 赛后总结、经验提取、简历素材
```

### 竞技赛道文件夹模板（待创建）

竞技赛道收敛为 **3 套模板**，覆盖所有竞技类比赛：

#### 模板A：纯刷题竞技（蓝桥杯 / ACM / 天梯赛）

├── 02-知识学习/        # 按知识模块分文件夹（算法/数据结构/...）
├── 03-刷题训练/        # 题目分类.md、代码解答/、错题本.md
├── 04-模拟实战/        # 模考记录.md、正确率统计.md、薄弱点诊断.md

项目根目录/

├── 进度追踪.md
├── 项目日志.md
├── 用户画像.md

#### 模板C：综合竞技（数模 / 电子设计 / 机械创新）

### 用户可定制

模板是起步时不迷路的引导，不是锁死的。用户可以：
- 删掉不需要的文件夹
- 新增自定义文件夹
- 重命名任何文件夹

### 公共文件层说明

无论哪个赛道，`项目信息.md`、`进度追踪.md`、`项目日志.md` 的结构保持一致，只有内容语义不同：
- 方案赛道记「方案写到第几章了」
- 竞技赛道记「刷了多少题」「模考正确率多少」

## 三、核心技术方案：上下文管理

### 问题

一个竞赛项目全流程可能产生 150+ 条消息。如果全在同一个对话里：

- AI 混淆早期指令和当前需求
- 输出质量下降
- Token 成本线性增长

### 解决方案：对话分段 + 交接包

```
一个项目 = 一个工作区
一个阶段 = 一个对话（上下文独立）
文件存储在项目文件夹（所有对话共享）
```

### 阶段划分

#### 方案赛道阶段

| 阶段        | 建议对话数 | 说明            |
| --------- | ----- | ------------- |
| 选赛        | 1 个对话 | 画像填写 + 匹配推荐   |
| 调研        | 1 个对话 | 信息收集 + 分析     |
| 方案设计-搭框架  | 1 个对话 | 基于往届作品搭框架     |
| 方案设计-写初稿  | 1 个对话 | AI+用户协作写稿     |
| 方案设计-审改迭代 | 1 个对话 | 审改循环（紧密循环，不拆） |
| PPT 路演    | 1 个对话 | 大纲 + Q&A      |
| 赛后沉淀      | 1 个对话 | 总结 + 经验提取     |

#### 竞技赛道阶段

| 阶段        | 建议对话数 | 说明            |
| --------- | ----- | ------------- |
| 选赛        | 1 个对话 | 画像填写 + 匹配推荐（共用） |
| 备赛规划      | 1 个对话 | 学习路线 + 资源清单 + 时间规划 |
| 知识学习      | N 个对话 | 按模块学习，随时提问，不限对话数 |
| 刷题训练      | N 个对话 | 分题型刷题，AI 启发式辅导   |
| 模拟实战      | 1 个对话 | 计时模考 + 诊断报告    |
| 赛后沉淀      | 1 个对话 | 总结 + 经验提取（共用）  |

> 竞技赛道的「知识学习」和「刷题训练」不限制对话数，因为这是长期持续的过程，用户随时开新对话继续。


### 交接包机制

**自动化部分：**

- AI 检测到阶段产出完成 → 判断阶段结束
- 自动生成结构化交接包 → 存入项目文件夹
- 提示用户复制启动指令

**人工部分：**

- 用户新开对话 → 粘贴指令 → 回车（10秒内完成）

**兜底：**

- AI 未识别到阶段完成时，用户手动输入 `/phase-handoff` 生成交接包

### 交接包数据结构

```markdown
# 阶段交接：[阶段名称]
- 完成时间
- 关键产出（文件路径列表）
- 重要决策（决策列表）
- 当前状态（一句话）
- 传递给下一阶段的数据（结构化）
- 下一阶段（名称 + 启动指令）
```

### 阶段间数据传递

#### 方案赛道

| 传递方向     | 传递内容                     |
| -------- | ------------------------ |
| 选赛 → 调研  | 确定的目标竞赛、匹配理由、竞赛类别、往届获奖规律 |
| 调研 → 方案  | 往届作品分析、市场调研、赛道竞争情况、差异化方向 |
| 方案 → PPT | 完整方案文档、核心亮点、数据支撑、创新点摘要   |
| PPT → 复盘 | PPT终稿、答辩记录、评委反馈、答辩表现     |

#### 竞技赛道

| 传递方向       | 传递内容                     |
| ---------- | ------------------------ |
| 选赛 → 备赛规划  | 确定的目标竞赛、考纲/考点、竞赛类别    |
| 备赛规划 → 知识学习 | 学习路线图、推荐资源清单、时间分配      |
| 知识学习 → 刷题训练 | 知识点掌握情况、薄弱模块标记         |
| 刷题训练 → 模拟实战 | 刷题量统计、错题分布、正确率趋势       |
| 模拟实战 → 复盘  | 模考成绩、薄弱点诊断、时间管理分析     |

## 四、用户画像设计

### 画像分层


| 层级      | 内容                           | 填写方式               |
| ------- | ---------------------------- | ------------------ |
| **基础层** | 学校、专业、年级、兴趣方向、目标（保研/就业/能力提升） | 一次填写，跨项目复用         |
| **项目层** | 技术栈能力、可用资源（企业/实验室）、团队情况、指导老师 | 每个项目独立填写或从旧项目 copy |


选赛阶段只需要基础层，方案设计阶段需要项目层的额外信息。

## 五、兜底机制设计

每个阶段都设计两套路径：


| 路径       | 触发条件          | 产出                  |
| -------- | ------------- | ------------------- |
| **深度路径** | 用户积极配合，提供充分信息 | 高质量产出               |
| **兜底路径** | 用户不参与/无法提供信息  | AI 独立完成，标注 ⚠️ 待用户确认 |


**原则：** 产品不能在用户不提供信息时卡住，必须能在信息不完整的情况下继续前进。

**可选步骤：** 用户可以选择跳过某些阶段（如已选好赛则直接进入调研），不需要强制走完所有流程。

## 六、提示词管理策略

### 原则

每个模块的子步骤都有对应的精心打磨的提示词。提示词是项目的核心竞争力之一。

### 调用方式

根据用户当前状态和项目进度，自动判断该调用哪个提示词。

### 保护策略

- 当前阶段：不花精力在「技术防泄露」上
- 真正的壁垒 = 独特流程方法论 + 精心打磨的提示词
- 即使别人拿到提示词文本，不理解流程设计也做不出同样效果

## 八、网页展示框架设计（比赛用）

### 页面结构

```
首页 → 选赛页 → 赛道分流 → 方案赛道流程（调研→方案→PPT）→ 复盘页 → 个人中心
                         ↘ 竞技赛道流程（备赛→学习→训练→模考）↗
```

每个页面：

- 真实 UI 布局（输入框、按钮、卡片、列表）
- 模拟交互效果（点击有反应、加载有动画）
- 假数据填充展示效果（不需要接真实 AI）
- 关键流程用弹窗/步骤条展示

### 技术选型建议

纯静态网页（HTML + CSS + JS），不需要后端。快速搭建，评委能看到产品长什么样即可。

## 十一、设计原则（持续遵守）

1. **便捷第一**：用户操作动机要非常小，上手难度要低
2. **不能卡住**：用户不提供信息时，产品也要能往前走
3. **分步写作**：写方案时一次一个章节，用户确认后再写下一部分
4. **数据铁律**：不编造任何数据、案例、引用，没有出处的信息标注 ⚠️ 待确认
5. **AI 评审后人拍板**：AI 可以模拟评委打分挑毛病，但最终判断必须人做
