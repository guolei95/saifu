## 三、第1步：CLI原型

> 时间：5 月 3 日 | 耗时：~5 小时 | 花费：¥0

### 为什么不直接做网页

**这是最重要的经验之一。** 不要一上来就写网页。先用最简单的方式验证想法对不对。

CLI（命令行界面）的好处：
- 没有样式布局乱七八糟的事（不用纠结按钮放哪、颜色对不对）
- 纯逻辑验证：搜索能不能搜到竞赛？AI 匹配有没有意义？
- 生成速度极快：AI 一次输出几百行 Python，复制粘贴就能跑

### 具体做了什么

跟 AI 说：「帮我写一个 Python 程序，用户在终端里填个人信息，然后调用搜索引擎找竞赛，再用 AI 匹配打分，最后打印结果。」

AI 生成了 `match_competitions.py`（约 2000 行 Python），功能：
1. 用户输入：专业、年级、兴趣方向、技能水平
2. DuckDuckGo 搜索：按关键词搜竞赛信息
3. DeepSeek AI 匹配：分析每条竞赛是否适合这个用户
4. 终端打印结果：排名 + 理由 + 风险提示

### 验证通过的标准

在终端里跑了几次，确认：
- ✅ 能搜到真实的竞赛
- ✅ AI 匹配的结果有意义（不是乱说的）
- ✅ 输出的格式清晰可读

**逻辑跑通了再往下走，不然做网页白费力气。**

## 四、第2步：拆成后端+前端

> 时间：5 月 7-8 日 | 耗时：~10 小时 | 花费：¥0

### 做了什么

把 2000 行的单文件 Python 拆成模块化的 Web 架构。

### 后端（Python FastAPI）— 6 个模块

```
saifu/backend/
├── main.py           ← FastAPI 入口，定义所有 API 接口
├── config.py         ← 配置文件（API Key 存这里）
├── match/engine.py   ← 匹配引擎，8 步流程
├── services/search.py       ← DuckDuckGo 搜索
├── services/ai_client.py    ← DeepSeek API 调用
├── services/knowledge_base.py ← 竞赛常识库（84 项 A 类竞赛数据）
└── services/validation.py   ← 交叉验证 + AI 自审查
```

### 前端（纯 HTML + CSS + JS）— 3 个文件

```
saifu/
├── index.html    ← 页面结构（表单 + 结果卡片 + 加载动画）
├── css/style.css ← 页面样式
├── js/app.js     ← 前端逻辑（收集表单 → 调 API → 渲染结果）
```

### 关键决策：为什么不用框架

| 方案 | 优缺点 |
|------|--------|
| React / Vue | 功能强大但复杂，AI 生成代码质量不稳定，对简单页面过度设计 |
| **原生 HTML/CSS/JS** ✅ | 简单直接，AI 生成准确率高，文件少好管理 |

**原则：功能简单就用最简单的技术栈。** 我们的前端就是一个表单 + 结果展示，不需要框架。

### 关键安全决策

> ⚠️ API Key（DeepSeek 的密钥）必须放在**后端环境变量**里，绝对不要写在前端 JS 代码里！
>
> 为什么：前端代码如果上传到 GitHub 公开仓库，全世界都能看到你的密钥，别人可以盗用。

## 五、第3步：前端部署到 Vercel

> 时间：5 月 8-9 日 | 耗时：~2 小时 | 花费：¥0

### 目标

让别人能打开一个网页看到你的网站，而不是只能在你电脑上看。

### 为什么选 Vercel

| 平台 | 免费 | 国内访问 | 自动部署 |
|------|------|---------|---------|
| Vercel ✅ | 是 | 香港节点，较快 | push 代码自动部署 |
| GitHub Pages | 是 | 慢 | 手动/自动 |
| 阿里云 OSS | 否（需备案） | 快 | 手动 |

### 具体操作

1. **注册 Vercel**：去 [vercel.com](https://vercel.com)，点「Sign Up」，选「Continue with GitHub」（用 GitHub 账号登录）
2. **导入项目**：在 Vercel 后台点「Import」→ 选择你的 GitHub 仓库
3. **设置根目录**：Root Directory 设为 `saifu`（你的前端文件在哪个文件夹）
4. **部署**：点 Deploy，等 2 分钟
5. **得到网址**：`https://saifu-75e4.vercel.app`（Vercel 自动生成的域名）

### Vercel 配置文件

在 `saifu/vercel.json` 里写：
```json
{
  "version": 2,
  "builds": [{ "src": "**", "use": "@vercel/static" }]
}
```

### 自动部署机制

之后每次改代码，只需要：
```bash
git add .
git commit -m "改了xxx"
git push
```
Vercel 检测到 push 后 **30 秒内自动部署**，网址不变。

## 六、第4步：后端用 Cloudflare Tunnel 穿透

> 时间：5 月 9 日 | 耗时：~3 小时（含踩坑） | 花费：¥0

### 问题

后端跑在你电脑上（`localhost:8000`），只有你能访问。怎么让别人在浏览器里调你的后端？

### 解决方案

**Cloudflare Tunnel**：把你电脑上的端口映射成一个公网 URL。免费。

### 具体操作

1. 下载 `cloudflared.exe`（Cloudflare Tunnel 客户端），放到 `C:\Users\你的用户名\`
2. 启动后端：
   ```bash
   cd saifu/backend
   python -m uvicorn main:app --host 0.0.0.0 --port 8000
   ```
3. **新开一个终端**（不要关掉后端的那个），启动 Tunnel：
   ```bash
   cloudflared.exe tunnel --url http://localhost:8000
   ```
4. 看输出日志，找到类似 `https://xxx.trycloudflare.com` 的 URL
5. 这个 URL 就是你的后端公网地址，**全世界都能访问**

### 注意事项

- ⚠️ 每次重启电脑，Tunnel URL 会变（新的随机域名）
- ⚠️ 免费版偶尔会断连（后面第 6 步会解决这个问题）
- 备选方案：`ssh -R 80:localhost:8000 nokey@localhost.run`（SSH 隧道，同样免费）

## 七、第5步：前后端对接

> 时间：5 月 9 日 | 耗时：~1 小时

### 做什么

让前端页面能调用到后端 API。

### 具体操作

1. 在 `saifu/js/app.js` 里找到 API 地址那一行：
   ```javascript
   const API_BASE_URL = "https://xxx.trycloudflare.com";
   ```
2. 改成你第 4 步得到的 Tunnel URL
3. 保存，推送到 GitHub：
   ```bash
   git add saifu/js/app.js
   git commit -m "fix: 更新Tunnel URL"
   git push
   ```
4. 等 30 秒，Vercel 自动部署完成

### 用户访问完整链路

```
用户打开 https://saifu-75e4.vercel.app
  → 看到表单页面（来自 Vercel，全球 CDN）
  → 填写信息，点「开始匹配」
  → 前端 JS 调用 https://xxx.trycloudflare.com/api/match
  → Tunnel 把请求转发到你电脑的 FastAPI（localhost:8000）
  → Python 后端处理：搜索 + AI 匹配 → 返回结果
  → 前端渲染成卡片展示
```

## 八、第6步：架构优化

> 时间：5 月 9 日 | 耗时：~3 小时

### 遇到的问题

匹配流程需要 **60-180 秒**（搜索 + 多轮 AI 调用），但 Cloudflare 免费 Tunnel **对超过 60 秒的连接极不稳定**，经常断开。

现象：用户等了两分钟，页面报「无法连接到服务器」，体验极差。

### 改造方案

从「单次长请求」改成「**提交 + 轮询**」模式：

```
❌ 旧方案（会断）：
  浏览器 ──POST一次请求──→ 等60~180秒 ──→ 返回结果
  中途Tunnel断开 → 全部丢失 → 用户重试 → 又断开

✅ 新方案（不断）：
  浏览器 ──POST提交画像──→ 立即返回 task_id（<1秒）
  浏览器 ──每2秒问一次：好了没？──→ processing / done
  单次轮询失败 → 自动重试，不受Tunnel波动影响
```

### 改了 3 个文件

| 文件 | 改动内容 |
|------|---------|
| `backend/main.py` | POST 立即返回 task_id + 后台异步处理 + GET `/api/match/{task_id}` 轮询接口 |
| `backend/services/ai_client.py` | 加 `timeout=120.0` 防 API 卡死 |
| `saifu/js/app.js` | 单次 fetch → 循环轮询，失败自动重试 |

`.asia` 是亚洲区域的顶级域名，适合面向亚洲用户的网站。和 `.com` 一样是国际域名，不受中国特殊限制（但用了阿里云，就得按国内规矩来）。

## 十、第8步：阿里云实名认证

> 时间：5 月 10 日 | 耗时：~1 小时操作 + 等待 ~8 小时审核 | 花费：¥0

### 这是国内买域名绕不过的一关

在中国大陆的域名注册商（阿里云、腾讯云等）买域名，**必须完成实名认证**才能正常使用。不然域名会被「暂停解析」—— 别人输入你的域名打不开网站。

### 流程

```
提交身份信息 → 阿里云初审（几分钟） → 注册局最终审核（~8小时） → 审核通过
```

### 具体操作

1. 在阿里云域名控制台 → **信息模板**
2. 新建一个「个人」类型模板
3. 填写：
   - 姓名（真实姓名）
   - 身份证号
   - 上传身份证正反面照片（手机拍照即可）
   - 邮箱（用你自己的，需要验证）
4. 阿里云会给你邮箱发验证链接 → 点一下完成邮箱验证
5. 提交审核

### 我遇到的问题

- 域名详情页显示「持有者实名认证：未实名认证」
- 邮箱显示 `zyhl-admin@list.alibaba-inc.com` 验证未通过

**这不是 bug！** 原因是：
- 买域名时阿里云自动给你分配了一个 **隐私保护邮箱**（`zyhl-admin@...`），不是你填的邮箱
- 你的信息模板虽然提交了（审核中），但域名还没有「过户」关联到这个模板
- 等模板审核通过 → 点「快速过户」→ 把域名挂到已认证的模板上 → 状态就正常了

### 审核需要多久

阿里云官方说「一般 1 天内，部分 3-5 个工作日」。实际上根据系统统计，**身份证审核约 7-8 小时**。

**经验：上午提交，晚上查；下午提交，第二天查。** 不用一直刷新。

## 十一、第9步：DNS 解析配置

> 时间：5 月 10 日 | 耗时：~10 分钟 | 花费：¥0

### DNS 是什么

DNS 就是「域名 → IP 地址」的翻译系统。你买的域名 `saifu.asia` 只是一个名字，需要告诉全世界「这个名字对应哪个服务器」。

### 操作

1. 阿里云控制台 → **域名列表** → 找到 `saifu.asia` → 点「解析设置」
2. 点「添加记录」：

| 字段 | 填什么 |
|------|--------|
| 记录类型 | CNAME |
| 主机记录 | @（代表根域名 `saifu.asia`） |
| 记录值 | `saifu.3359717058.workers.dev`（Cloudflare Pages 给的临时域名） |
| TTL | 默认（10 分钟） |

3. 保存

### ⚠️ 注意

- 如果阿里云提示「CNAME 不能与 @ 共存」：改用 `www` 做主机记录，然后开 URL 转发把根域名跳到 `www`
- DNS 修改后不是立刻生效的，全球传播可能需要几分钟到几小时
- **但域名必须先通过实名认证**，不然 DNS 配置了也不生效

## 十二、第10步：Cloudflare Pages 绑定自定义域名

> 时间：待完成 | 预计耗时：~15 分钟 | 花费：¥0

### 为什么还需要这一步

DNS 的 CNAME 记录只是告诉全世界「`saifu.asia` → Cloudflare Pages」，但 **Cloudflare Pages 不知道 `saifu.asia` 是你的网站**。需要在 Pages 后台加一次「自定义域」，两边握手成功。

### 操作（待实测）

1. 登录 [Cloudflare Dash](https://dash.cloudflare.com/)
2. 进入 Pages 项目 → 点「自定义域」
3. 添加 `saifu.asia`
4. Cloudflare 会验证 DNS 记录 → 验证通过 → 绑定成功
5. 等待 DNS 全球传播（几分钟到几小时）
6. 浏览器输入 `saifu.asia` → 🎉

### 这一步做完之后

你的网站就可以通过 `saifu.asia` 访问了。

> ⚠️ **关于国内访问**：Cloudflare Pages 默认走全球节点（香港/日本/新加坡），国内用户能访问但速度不如国内服务器快。追求极致速度需要 ICP 备案 + 国内服务器部署，这是后续的事。

## 十四、踩过的 10 个坑

### 坑 1：不要一上来就做网页 ✅ 已踩已反思

先做 CLI 原型验证逻辑，确认能跑通再拆成网页。省了至少一整天无效折腾。

### 坑 2：Tunnel 长请求会断 ✅ 已解决

Cloudflare 免费 Tunnel 对超过 60 秒的连接极不稳定。解决方案：提交+轮询模式（见第 6 步）。

### 坑 3：Tunnel URL 重启就会变 ✅ 部分解决

每次电脑重启，Tunnel 给的新 URL 都不一样，需要手动更新 `app.js` 里的地址并重新部署。买了域名后这个问题消失了（域名永远不变）。

### 坑 4：前端文件目录结构折腾 ✅ 已踩已反思

为了适配 Cloudflare Pages，把 `frontend/` 子目录的文件移到根目录。后来发现没必要，又要求移回去。折腾了两轮。
**教训**：部署前先想清楚目录结构，改一次就够。

### 坑 5：Vercel 国内被墙 ❌ 无解

Vercel 的域名在国内某些网络下打不开。换到了 Cloudflare Pages（国内有边缘节点，访问更稳定）。

### 坑 6：DeepSeek API 余额不够 ⚠️ 偶发

免费额度有限，充 10 块钱够用很久。建议在 API 后台设一个余额提醒。

### 坑 7：搜索词硬编码导致结果不准确 ✅ 已解决

最初搜索词是写死的 8 条 STEM 关键词，对不同专业不友好。改为按用户专业方向动态生成搜索词，准确性大幅提升。

### 坑 8：去重逻辑太保守 ✅ 已解决

同名竞赛因为来源不同（官网/报名页/资讯页）被当成多条不同结果。重写了归一化逻辑，URL 三重去重。

### 坑 9：域名实名认证 ≠ 域名过户 ⚠️ 教训

信息模板通过实名审核后，还需要把域名「过户」关联到模板，域名才会显示「已实名」。很多人卡在这一步不理解为什么模板通过了域名还是未实名。

### 坑 10：阿里云隐私邮箱的困惑 ⚠️ 教训

域名详情页显示的 `zyhl-admin@list.alibaba-inc.com` 是阿里云的隐私保护邮箱，不是你填错的。过户到你的模板后会变成你自己的邮箱。

## 十五、给零基础新手的 7 条建议

### 1. 先跑通再优化

你的第一个版本不需要完美。功能跑通了，再去改样式、改体验。不要在前 10% 的进度上花 80% 的时间。

### 2. 把 AI 当你的程序员搭档

你不会写代码没关系，你会描述需求就行。

❌ 不好的描述：「帮我做一个网站」
✅ 好的描述：「帮我写一个网页，上面有一个表单，用户填姓名、专业、年级，点提交后调用后端 API，返回的结果用卡片展示，每条卡片包含竞赛名称、适合程度、理由」

描述越具体，AI 产出越准。

### 3. API Key 绝对不能放前端

这是血的教训。DeepSeek/OpenAI 的密钥放在前端 JS 代码里 = 公开在 GitHub 上 = 全世界都能盗用你的额度。

**正确做法**：放后端环境变量（`.env` 文件），`.env` 加入 `.gitignore`。

### 4. 免费服务够用了

对于学生项目/个人网站，Vercel + Cloudflare Tunnel + DeepSeek 免费额度 + 便宜域名，完全够用。不需要一上来就买服务器。

### 5. 部署比写代码更容易踩坑

写代码有 AI 帮你，但部署涉及注册各种平台、配置环境变量、调 DNS，这些 AI 不能直接帮你点按钮。**部署花的时间可能比写代码还多，提前做好心理准备。**

### 6. 域名早点买

一个自己的域名 ¥8/年，能解决很多问题：不用每次重启改地址、给别人发链接不尴尬、看起来专业。早买早省心。

### 7. 记录过程，积累经验

我在做的：每做完一步就写下来（就是你现在看到的这篇教程）。以后面试、写简历、跟人分享都用得上。AI 写代码容易，但「从零到上线」的完整经历才是稀缺的。
