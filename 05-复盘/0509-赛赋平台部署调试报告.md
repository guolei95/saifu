# 赛赋平台部署调试报告

> 2026-05-09 | 电脑重启后的全链路恢复 + 架构优化记录

---

## 一、背景

电脑重启后，赛赋平台所有服务停止。需要恢复：后端（localhost:8000）+ Cloudflare Tunnel + Vercel 前端。

---

## 二、恢复步骤（标准化流程）

### 2.1 启动后端

```bash
cd saifu/backend
python -m uvicorn main:app --host 0.0.0.0 --port 8000
```

### 2.2 启动 Cloudflare Tunnel

```bash
# cloudflared.exe 位于 C:\Users\guolei\
cloudflared.exe tunnel --url http://localhost:8000
```

启动后日志中会出现新 URL，例如：`https://xxx.trycloudflare.com`

### 2.3 更新前端 API 地址

编辑 `saifu/frontend/js/app.js` 第 10 行，将 `API_BASE_URL` 改为新的 Tunnel URL。

### 2.4 推送部署

```bash
git add frontend/js/app.js
git commit -m "fix: 更新Tunnel URL"
git push
```

Vercel 自动部署（约 30 秒生效）。

## 三、遇到的问题与解决

### 问题1：cloudflared 丢失

**现象**：`where cloudflared` 找不到

**原因**：电脑重启后，之前安装的 cloudflared.exe 被清理

**解决**：重新下载到 `C:\Users\guolei\cloudflared.exe`（注意不能下到 OneDrive 同步目录，权限会拒绝）

---

### 问题2：匹配超时（120秒不够）

**现象**：前端报「匹配超时（超过 120 秒）」

**原因**：匹配流程包含搜索（~17 条查询）+ 3-4 轮 DeepSeek API 调用，总耗时 60-180 秒

**解决**：
- 前端超时 120s → 300s
- 搜索查询数 25 → 17 条
- 单次搜索结果 8 → 5 条
- 查询间隔 0.8s → 0.3s
- LLM max_tokens 16384 → 8192
- LLM 要求结果数 12+ → 6+

---

### 问题3：Tunnel 频繁断开（1033 错误）

**现象**：前端报「无法连接到服务器」（Failed to fetch），Tunnel 返回 Cloudflare 1033 错误

**原因**：Cloudflare Quick Tunnel 无 SLA 保障，进程重启后 URL 失效

**解决**：每次重启电脑后必须重建 Tunnel（URL 会变），更新 app.js 并推送

---

### 问题4：后端进程卡死

**现象**：localhost:8000 端口在监听但无响应（health 超时）

**原因**：
1. 匹配请求处理时间过长（60-180s）
2. 期间用户重试发送新请求，多请求堆积
3. uvicorn 单 worker 无法处理并发，进程阻塞

**解决**：
- 每次后端卡死后：`taskkill /F /PID <pid>` → 重启
- 长期：架构改造（见问题5）

---

### 问题5（根因）：长连接不兼容 Tunnel（架构改造）

**根因分析**：

```
原架构：浏览器 ──POST /api/match──→ Tunnel ──→ 后端 ──60~180s──→ 返回
         ↑                                                     ↑
         单次 HTTP 请求等到底，中途断开就全丢
```

- 匹配流程 60-180 秒是刚需（搜索 + 多轮 AI 调用无法省略）
- Cloudflare Quick Tunnel 对 >60s 的长连接极不稳定
- 一旦连接断开，用户只能重试，重试又加重后端负载

**架构改造：提交 + 轮询模式**

```
新架构：浏览器 ──POST /api/match──→ Tunnel ──→ 后端 ──→ 返回 task_id（<1s）
              ↑
              每 2s 轮询 GET /api/match/{task_id} ──→ 返回 processing/done
              ↑
              最大 300s，单次轮询失败自动重试
```

**改动文件**：

| 文件 | 改动 |
|------|------|
| `backend/main.py` | POST 立即返回 task_id + 后台 asyncio.create_task + GET 轮询接口 + 任务清理 |
| `backend/services/ai_client.py` | OpenAI 客户端加 `timeout=120.0` 防卡死 |
| `frontend/js/app.js` | 单次 fetch → submit+while轮询，轮询失败自动重试 |

**测试结果**：任务 `6afee909`，58 秒完成，返回 6 条竞赛推荐 ✅

## 五、已知局限

| 局限 | 影响 | 对策 |
|------|------|------|
| Tunnel 临时 URL 重启后变化 | 每次重启需手动更新 app.js + git push | 已形成标准化流程 |
| Tunnel 无 SLA | 偶发断连，Cloudflare 500 错误 | 轮询模式自动重试，不丢任务 |
| 本地后端依赖小雷电脑 | 电脑关机则服务不可用 | 后续考虑 Render 免费档 |
| Quick Tunnel 有速率限制 | 高并发时可能被限 | 当前学生规模够用 |
