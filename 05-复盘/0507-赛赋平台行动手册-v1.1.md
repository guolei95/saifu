## 一、项目背景（给 AI 读的上下文）

### 1.1 我们在做什么

把现有的 Python CLI 竞赛匹配脚本（`0503-match_competitions.py`）改造成一个**在线网页平台**，让零基础用户打开浏览器就能用。

### 1.2 核心原则

- **用户零门槛**：打开网址 → 填画像 → 点按钮 → 看结果。不需要安装、不需要 API Key、不需要配置环境
- **后端持有 API Key**：DeepSeek Key 只在服务器环境变量里，前端代码里不能出现
- **架构可扩展**：现在只做匹配，但代码结构要能以后加竞赛浏览、调研库、方案助手
- **面向中国用户**：前端放 Vercel（香港节点），后端放 Railway（美国），延迟可接受

### 1.3 项目文件路径

```
工作区根目录：D:\我的竞赛项目-AI赋能竞赛系统\

  - D:\我的竞赛项目-AI赋能竞赛系统\02-方案\0503-match_competitions.py  ← 现有匹配逻辑（约2000行）
  - D:\我的竞赛项目-AI赋能竞赛系统\03-素材\用户画像表单.html              ← 现有前端表单

  - D:\我的竞赛项目-AI赋能竞赛系统\项目日志.md                            ← 日志
```

### 1.4 现有代码的核心功能（AI 要先读懂）

`0503-match_competitions.py` 做的事：
1. 接收用户画像（学校、专业、年级、兴趣、技能等，约 25 个字段）
2. 根据画像生成搜索查询词（`generate_search_queries`）
3. 用 DuckDuckGo 搜索竞赛信息（`search_competitions`）
4. 用 DeepSeek 匹配分析（`match_and_enrich`：打分、匹配原因三段式、避坑提醒）
5. 用内置常识库交叉验证（`enrich_with_facts`：修正 URL、时间、费用）
6. L1 跨搜索结果交叉验证（`cross_source_verify`：日期一致性、URL 来源检查）
7. L2 DeepSeek 自审查（`self_review_results`：让 LLM 回顾自己的输出挑错）
8. 输出格式化结果（18+ 字段/每条竞赛）

## 二、技术选型（已确定，不要改）

| 层 | 技术 | 理由 |
|----|------|------|
| 后端框架 | **FastAPI**（Python） | 轻量、能把现有代码几乎原封不动变 API |
| 前端 | **纯 HTML + CSS + JavaScript** | 你现有表单已经是 HTML，直接扩展，不推翻 |

| 前端部署 | **Vercel**（vercel.com）免费档 | 有香港节点，国内访问快 |
| AI 服务 | DeepSeek API（你现在的 Key） | 不变 |

## 七、技术注意事项（提前避坑）

### ⚠️ 注意 1：API 调用超时
匹配过程需要 30-60 秒。前端 fetch 要设置 120 秒超时。后端不需要特殊处理。

### ⚠️ 注意 2：搜索限速
20+ 条查询，每条间隔 0.8 秒 = 约 16 秒搜索时间。不要并发出太多以免被封。

### ⚠️ 注意 3：JSON 解析容错
DeepSeek 返回的 JSON 经常带 markdown 代码块或格式瑕疵。ai_client.py 必须参考原代码的多种修复策略。

### ⚠️ 注意 4：HTTPS 混用
本地测试前端用 `http://localhost`，部署后 Vercel 是 HTTPS，Railway 也是 HTTPS。前端 JS 要能切换。

### ⚠️ 注意 5：字段名兼容
原代码字段名有新旧两种（如 `match_score` / `s`、`match_reason` / `r`）。engine.py 中统一用新字段名（完整名称），不再兼容旧缩写。

### ⚠️ 注意 6：不要用 async
原代码全是同步的。不要强行改 async。FastAPI 的 `def`（非 async）会自动在线程池中运行，不会阻塞。
