## 🔴 致命错误（不改就跑不起来）

### 1. config.py — DeepSeek API 地址少了 `/v1`

```diff
- DEEPSEEK_BASE_URL = "https://api.deepseek.com"
+ DEEPSEEK_BASE_URL = "https://api.deepseek.com/v1"
```

**后果**：OpenAI SDK 会在 base_url 后面拼接 `/chat/completions`。  
正确路径：`https://api.deepseek.com/v1/chat/completions`  
错误路径：`https://api.deepseek.com/chat/completions` → **404 报错**。

实际代码第44行写的就是 `/v1`，手册抄错了。

---

### 2. engine.py — profile 格式严重缩水

手册里写的 profile 只有 8 个字段：
```python
{"school", "major", "grade", "interests", "skills", "prefer_type", "prefer_level", "available_time"}
```

但实际代码（`match_and_enrich` 函数，第1047-1068行）构建的 `profile_text` 用了 **25 个字段**：
```
school, major, grade, interests, skills, tech_directions, goals, time_commitment,
summer_winter, preference, team_preference, has_advisor, can_cross_school, avoid_types,
past_national, past_provincial, past_school, representative_projects, preferred_duration,
preferred_format, fee_budget, language_pref, preferred_team_size, ...
```

**后果**：如果 engine.py 只收 8 个字段，`match_and_enrich` 里的大量字段会是空值，LLM 匹配质量严重下降。

## 🟢 小问题（不影响核心功能）

### 9. async vs sync 混淆

手册要求 `async def match_competitions(...)`，但现有代码全部是同步的。DuckDuckGo 搜索和 DeepSeek API 调用都不是异步的。

**实际做法**：保持同步函数，在 FastAPI 中用 `def`（不是 `async def`），FastAPI 会用线程池自动处理。或者如果真的想并行搜索，用 `concurrent.futures.ThreadPoolExecutor`。

---

### 10. 反馈纠正功能没提

原代码有完整的用户纠错持久化系统（`save_correction`、`load_corrections`、`apply_corrections`，第816-875行），把纠正信息存到本地 JSON。手册完全没提这个功能。

**影响**：V1 可以暂时跳过，但后续应该加上。

---

### 11. `parse_profile_template` 函数没提及

原代码有一个从 Markdown 模板文件中解析用户画像的函数（第334-480行，158行代码）。手册的前端部分是直接从网页表单收集数据，所以理论上不需要这个函数了。**但这个函数有复杂的字段映射逻辑，前端收集数据时可以参考。**

---

### 12. 缺少 `services/__init__.py`

手册只有 `match/__init__.py`，没有 `services/__init__.py`。对 Python 3.3+ 不影响运行，但为了代码规范应该加上。

---

### 13. Vercel 配置已过时（不影响运行）

手册的 `vercel.json` 用的是 V2 API 格式。Vercel 现在推荐零配置（自动检测静态站点），不需要 vercel.json。不过写了也不会报错。

## 🔧 过程中会遇到的技术问题（提前预警）

### 问题1：API 调用超时

用户的匹配请求从发起到返回需要 **30-60秒**。浏览器默认的 fetch 超时可能不够。

**解决**：前端 fetch 设置较长超时（120秒），或者后端用 SSE 推送进度。

### 问题2：搜索可能被限速

原代码搜索间隔 0.8 秒，20+ 条查询需要约 16 秒搜索时间。

**解决**：考虑用 `ThreadPoolExecutor` 并行搜索（但要控制并发数防封）。

### 问题3：HTML 表单 → profile dict 的字段映射

现有 `用户画像表单.html` 生成的 Markdown 格式字段名，和 `engine.py` 需要的字段名，需要做映射。

**解决**：在 `app.js` 里直接按 engine.py 需要的字段名来收集数据，不经过 Markdown 中间格式。

### 问题4：HTTPS 混用问题

本地测试时前端用 `http://localhost`，部署到 Vercel 后是 `https://xxx.vercel.app`，后端 Railway 是 `https://xxx.railway.app`。前端调用后端必须是 HTTPS（因为 Vercel 是 HTTPS）。

**解决**：前端 JS 检测环境自动切换 API 地址。
